# 第一轮朋友试用清单

## 试用入口

https://18ok.github.io/daily-fitness-plan/

本轮只邀请 3～5 位平时主要使用手机、对健身不熟悉的女性朋友。请她们使用自己的手机和常用浏览器，不要先讲解页面怎么操作。

## 可直接发送的话术

> 我做了一个给健身新手用的“今日计划”小工具，想请你用手机试一次。打开后按你今天的真实情况选择，找到今天的计划并确认就可以。过程中先不要问我怎么点；如果哪里看不懂、觉得麻烦或不符合实际，请直接告诉我。整个过程大约 1 分钟。

## 每位用户只观察这 6 项

| 项目 | 记录方式 |
| --- | --- |
| 是否独立完成 | 是 / 否；若否，卡在哪里 |
| 首次看到并确认计划的时间 | 秒；目标不超过 60 秒 |
| 是否找到“调整” | 是 / 否 |
| 是否理解三个状态标签 | 全部理解 / 部分理解 / 不理解 |
| 双指放大后能否缩小 | 能 / 不能；记录手机和浏览器 |
| 计划是否符合真实一天 | 适合 / 太难 / 不符合状态；补一句原因 |

不要边看边教，也不要问“你喜不喜欢这个颜色”。只记录真实行为、卡点和原话。

## 第二天回访

只问三个问题：

1. 今天还会不会自己再打开？为什么？
2. 训练和饮食里，哪一条最难理解或最难做到？
3. 如果只能增加一个动作演示，你最希望先看到哪个动作或器械？

## 第一轮通过条件

- 至少 4/5 的用户无需口头帮助即可完成。
- 首次确认计划的中位时间不超过 60 秒。
- 所有实测手机均能双指放大并正常缩小。
- 至少 4/5 的用户能找到“调整”并理解当天状态标签。
- 没有同一个严重卡点出现在 2 人以上。

## 进入方案 2 的门槛

第一轮先不直接加入底部调整层或更重的互动。只有出现以下任一情况，才启动方案 2 的原型：

- 2 人以上找不到“调整”；
- 2 人以上在修改状态时来回滚动或误操作；
- 返回用户确认计划的中位时间超过 15 秒；
- 用户明确需要“先看到变化、再决定是否确认”。

若达到门槛，方案 2 只复用现有三个状态字段和计划生成逻辑，把调整操作放入移动端底部层，并增加即时预览；不改变页面信息结构，也不重写训练、饮食和记录规则。

## 本轮之后怎么改

将问题按出现人数排序，每轮只修复前 2～3 个高频问题。动作演示和器械科普按用户点名频率排序，等简笔动画素材能力就绪后再接入真实内容，不放空入口或占位视频。

## 当前已知限制

页面内反馈在 Supabase 表尚未部署时会安全保存在用户本机，联网且表可用后自动重试。正式收集线上反馈前，需要在 Supabase SQL Editor 执行 `supabase/experience_feedback.sql` 并验证插入权限。
